# 가상 스레드 400개를 켰을 때 p99에 무슨 일이 생기는가

> **실험일**: 2026-06-18 | **JDK**: Amazon Corretto 21.0.6 | **시리즈**: JVM 가상 스레드 실증 실험 #2

---

가상 스레드 400개를 띄웠을 때 p50은 12ms였고 p99는 4,704ms였습니다.

같은 쿼리, 같은 서버, 같은 코드. 스레드 모델만 달랐습니다. 모니터링 대시보드는 "평균 응답시간 12ms"를 보여줬고, p99 그래프만 조용히 4초를 넘어서고 있었습니다.

가상 스레드가 10ms짜리 쿼리를 5초짜리로 만들 수 있을까요? 실험을 통해 그 이유를 이론으로 예측하고 수치로 확인했습니다.

---

## 배경: 가상 스레드 전환의 약속

가상 스레드 도입에 앞서 [실험 #1 — NMT 메모리 분리](./2026-06-29-virtual-threads-nmt.md)에서 한 가지를 먼저 측정했습니다.

플랫폼 스레드 1,500개 vs 가상 스레드 10,000개를 같은 머신에 띄웠을 때 OS 스레드 수가 어떻게 달라지는지 확인한 것입니다.

```
플랫폼 스레드 1,500개 → OS 스레드 1,532개  (네이티브 스택 441,712 KB)
가상 스레드  10,000개 → OS 스레드    40개  (네이티브 스택  72,320 KB)
```

생성 시간도 달랐습니다. 플랫폼 1,500개는 80ms, 가상 10,000개는 20ms였습니다. 가상 스레드가 6.7배 더 많았는데 4배 빨리 만들어진 셈입니다.

이유는 구조적 차이입니다. 플랫폼 스레드는 생성할 때마다 OS에 `pthread_create` 시스콜을 날리고 네이티브 스택을 예약합니다. 가상 스레드는 JVM 힙 위의 자바 객체입니다. OS를 거치지 않아서 빠르고 가볍습니다.

숫자를 정리하면 이렇습니다.

| 항목 | 플랫폼 스레드 | 가상 스레드 |
|------|-------------|------------|
| 생성한 스레드 수 | 1,500 | 10,000 |
| 실제 OS 스레드 수 | 1,532 | **40** |
| 네이티브 스택 (per-thread) | 278.7 KB | **3.5 KB** |
| 힙 사용량 (per-thread) | 2.9 KB | 2.4 KB |

"그렇다면 가상 스레드를 많이 만들수록 처리량도 비례해서 오르지 않을까?"

이 질문이 실험 #2의 출발점이었습니다.

---

## 실험 설계

### 목표

커넥션 풀 크기와 가상 스레드 수를 동시에 바꿔가며 p50/p95/p99/max를 측정합니다.

가설은 하나입니다.

> **B-1: 풀이 작으면 가상 스레드 수를 늘렸을 때 p99가 지수적으로 폭발한다**

### 실험 환경

| 항목 | 값 |
|------|----|
| JDK | Amazon Corretto 21.0.6 |
| DB | H2 in-memory (2.2.224) |
| 커넥션 풀 | HikariCP 5.1.0 |
| 쿼리 지연 | 10ms (`H2 SLEEP` 함수로 실제 I/O 모사) |
| 총 작업 수 | 2,000 ops |
| 변수 | poolSize ∈ {5, 50} × vthreads ∈ {10, 50, 100, 200, 400} |

쿼리 지연을 10ms로 고정한 이유는 실제 DB I/O를 모사하면서도 재현 가능한 환경을 만들기 위해서입니다. `H2 SLEEP` 함수가 실제 네트워크 레이턴시 없이 지정 시간만큼 대기하므로, 커넥션 풀 경합만 격리해서 측정할 수 있습니다.

### 이론 예측

측정 전에 이론값을 먼저 계산했습니다. pool=5로 고정하고 vthreads를 늘리면 어떻게 돼야 할까요?

```
실질 처리량(ops/s) = pool_size × (1,000ms / query_ms)
                  = 5 × 100
                  = 500 ops/s

가상 스레드 400개, pool 5개일 때 대기 큐 깊이:
  큐 깊이 = vthreads - pool_size = 395
  이론 평균 대기 = (큐 깊이 / 실질 처리량) × 1,000ms
                = (395 / 500) × 1,000
                ≈ 790ms
```

이 790ms는 평균 대기 시간입니다. p50과는 다릅니다. p50은 커넥션을 빨리 잡은 태스크들의 중앙값이고, 대기 줄 끝에 선 태스크들의 지연이 p95~p99에 쌓입니다.

"평균은 괜찮지만 꼬리가 폭발"하는 전형적인 패턴을 이론상 예측할 수 있었습니다.

## 세 가지 발견

### 발견 1: p50은 거짓말한다

poolSize=5, vthreads=400에서 p50=13ms입니다.

"쿼리가 10ms니까 괜찮다"는 결론을 내릴 수 있습니다. 실제로 p99=4,704ms인 사용자들이 이미 타임아웃을 경험하고 있는 상황인데도 말이죠.

혹시 p50 또는 평균만 보고 있지는 않으신가요? 가상 스레드 전환 이후라면 p99/p95를 함께 보는 것을 권합니다. 이 실험에서 p50은 vthreads 수와 완전히 무관하게 일정했습니다. 변화는 꼬리에서만 일어났습니다.

### 발견 2: 가상 스레드 수는 실질 처리량과 무관하다

pool=5일 때 vthreads를 10→400으로 40배 늘렸습니다. 처리량이 올랐을까요?

```
pool=5, vthreads=10  → max 처리량 ≈ 500 ops/s (이론치)
pool=5, vthreads=400 → max 처리량 ≈ 500 ops/s (동일)
```

가상 스레드 400개 모두 실행 가능한 상태지만, DB 커넥션은 5개뿐입니다. 실제로 동시에 쿼리를 날릴 수 있는 스레드는 5개입니다. 나머지 395개는 커넥션을 기다리면서 대기 줄에 쌓입니다.

가상 스레드가 OS 스레드 병목을 해결했지만, 커넥션 풀이 새 천장이 됩니다.

### 발견 3: 병목 이동은 이론으로 예측 가능하다

실측값과 이론값이 정확히 맞았습니다.

poolSize=5, vthreads=400에서 이론 평균 대기 ≈ 790ms, 실측 p99=4,704ms. HikariCP 기본 `connectionTimeout=30s`에 수렴하는 패턴도 측정 데이터에 그대로 나타납니다. vthreads=200(4,622ms)와 400(4,704ms) 사이의 차이가 줄어드는 것이 그 증거입니다.

---

## 왜 가상 스레드는 이 문제를 만드는가

플랫폼 스레드였다면 이 문제가 어떻게 달랐을까요?

플랫폼 스레드는 OS 스레드와 1:1입니다. 스레드를 400개 만들려면 OS 스레드 400개가 필요합니다. 메모리 부족, OS 스레드 한계, 컨텍스트 스위칭 비용이 먼저 장벽이 됩니다. 실질적으로 수백 개 이상을 운영하기 어렵습니다.

가상 스레드는 이 장벽을 없앴습니다. 400개를 만들어도 OS 스레드는 캐리어 스레드 ~17개만 씁니다. 그래서 400개를 쉽게 만들 수 있습니다. 그런데 커넥션 풀은 그대로 5개입니다.

```
플랫폼 스레드 시대: OS 스레드 한계 때문에 400개를 만들기 어려움
                   → 커넥션 경합이 극단적으로 커지지 않음

가상 스레드 시대:  OS 스레드 한계 없음 → 400개를 손쉽게 만듦
                   → 커넥션 풀 5개에 400개가 몰림
                   → p99 폭발
```

가상 스레드가 메모리 병목을 없애면서, 그다음 병목인 커넥션 풀이 드러납니다.

## 해결: pool size를 스레드 수에 맞게 조정하기

poolSize=50으로 올리면 어떻게 됩니까?

| vthreads | poolSize=5 p99 | poolSize=50 p99 | 개선 |
|----------|--------------|----------------|------|
| 400 | 4,704ms | 781ms | **6x** |
| 200 | 4,622ms | 784ms | **5.9x** |
| 100 | 3,718ms | 552ms | **6.7x** |

poolSize=50, vthreads=400에서 p99=781ms입니다. 여전히 lathency가 있는데, 이유는 50개 커넥션에 400개가 경합하기 때문입니다. 완전한 해결을 원한다면 pool size ≥ 동시 요청 수여야 합니다.

실무에서의 권고는 두 가지입니다.

**1. pool size를 예상 최대 동시 DB 접근 수로 설정하기**

```
maxPoolSize = 예상 최대 동시 DB I/O 요청 수
```

가상 스레드를 쓴다고 pool size를 기존 그대로 두면 이 실험과 같은 상황이 됩니다.

**2. p99를 모니터링하기**

p50은 이 문제를 감지하지 못합니다. HikariCP 메트릭(`hikari.pool.Wait`) 또는 애플리케이션 레벨 p99/p95를 확인하세요. 커넥션 대기 시간이 쿼리 실행 시간보다 길다면 pool size 문제입니다.

## 다음: 가상 스레드의 또 다른 천장

커넥션 풀 문제를 해결했다고 끝이 아닙니다. 가상 스레드에는 또 다른 천장이 있습니다.

`synchronized` 블록 안에서 blocking I/O가 발생하면 가상 스레드가 캐리어 스레드에 고정(pinned)됩니다. JDK 21 기준으로 이 현상은 처리량을 **4.33배** 감소시킵니다.

```
[synchronized 실험 — JDK 21]

[synchronized]   913ms  throughput=219 ops/s   ← 이론 최악에 근접
[ReentrantLock]  211ms  throughput=948 ops/s

RatioSync/Lock = 4.33x → PINNING CONFIRMED
```

좋은 소식은 JDK 25(JEP 491)에서 `synchronized` pinning이 완전히 해결됐다는 것입니다. 나쁜 소식은 JNI 호출에서의 pinning은 JDK 25에서도 그대로라는 것입니다.

다음 글에서는 `synchronized` vs JNI pinning을 JDK 21/25에서 비교 측정하고, carrier 스레드 수 튜닝으로 JNI pinning을 얼마나 회복할 수 있는지를 다룹니다.

## 정리

가상 스레드가 OS 스레드 병목을 없애면, 그 아래에 있던 병목이 수면 위로 올라옵니다. 이 실험에서는 커넥션 풀이 그 역할을 했습니다.

핵심만 정리하면 다음과 같습니다.

| 상황 | 결과 |
|------|------|
| poolSize=5, vthreads=400 | p50=13ms, **p99=4,704ms** |
| poolSize=50, vthreads=400 | p50=12ms, **p99=781ms** |

가상 스레드를 쓴다면 HikariCP pool size를 반드시 재검토하세요. p50이 정상이어도 pool size가 병목이라면 사용자 일부는 이미 타임아웃을 경험하고 있습니다.
